# Secure Workspace Web Boundary

`routes.js` owns the OpenChamber web/server boundary for secure workspace policy, provider validation, reconciliation, cleanup, structured export artifacts, and host apply. `policy.js` maps validated persisted settings to the exact plugin `readPolicy` option shape. `structured-artifact.js` owns artifact validation, selection, conflict detection, locking, journaling, rollback, and recovery.

## Trust And Identity

- `createWorkspaceProviderOperations` is instantiated only on demand with policy from persisted settings and `sourceDirectory` from a validated persisted project. Browser policy, source-directory, image, context, and provider metadata never configure provider operations.
- The operations entrypoint is lazy-loaded. A pinned package without `./operations` does not prevent server startup; provider validation, cleanup, and export return an explicit unavailable response until a compatible package is pinned. Tests inject the operations factory directly.
- Workspace records are loaded through the generated OpenCode v2 SDK workspace list method.
- The authoritative workspace ID, project ID, and provider come from the OpenCode record. Versioned metadata must bind that record to its immutable provider resource ID; provider operations independently enforce resource ownership. A `syncList` recovery ID mismatch is accepted only through trusted `operations.adoptWorkspace`, which verifies the old metadata against provider resources and locked plugin state before rebinding the current OpenCode ID. Provider resource, project, provider, and original audit identity remain immutable.
- Create generates a cryptographically random provisional ID server-side and passes it to the official SDK. A failed create lists authoritative rows and compensates only that exact ID: complete provider cleanup precedes official removal, while partial cleanup preserves the row with explicit remaining resources. Successful create is returned only after bounded official status polling observes `connected`; polling timeout returns HTTP 202 with a retryable provisional `connecting` record.
- Cleanup calls `operations.cleanupWorkspace` first. The official SDK remove is called only after `{ ok: true, remainingResources: [] }`; partial cleanup is retryable and leaves the authoritative record intact.
- Explicit reconciliation calls `operations.reconcileWorkspace` with the authoritative record, persisted policy, bound `workspace.admin` reauthentication, and returns provider diagnostics. It is distinct from OpenCode `syncList`.
- Export calls `operations.exportWorkspace`. Only structured artifact v1 is accepted. Identity, provider, project, canonical target directory, timestamps, integrity, paths, entry metadata, binary flags and blob bytes, blob hashes/sizes, and byte/count limits are validated before the artifact is cached server-side. Text hunk coordinates, counts, context-hash shape, bounds, ordering, non-overlap, baseline slices, result slices, and exact full-result reproduction are also validated. Empty, expired, malformed, oversized, or mismatched artifacts are rejected.
- The browser receives a bounded review model, never artifact blobs, and can submit only server-issued file/hunk IDs. There is no raw patch summary, browser-content apply, or patch fallback route.
- Structured artifacts are stored as their exact JSON bytes below the private OpenChamber data directory at `workspace-exports/`. The directory is forced to mode `0700` and artifact files to `0600`; writes use a same-directory temporary file, file fsync, atomic rename, and directory fsync where supported. Cache startup removes prior-process regular artifact files, so exports never survive a server restart. Symlinked roots and unsafe entries fail closed without traversal.
- The artifact cache defaults to a 60-minute maximum lifetime, also honors earlier artifact expiration, and is bounded by 20 entries and 80 MiB of actual stored bytes. Expired and least-recently-used entries are deleted. Reads reopen and reparse the stored bytes, including integrity and expiration validation, rather than trusting retained decoded state; corruption and storage failures are explicit errors.
- Artifact download revalidates the live workspace identity and expiration, then returns the exact validated stored JSON bytes with `no-store` and a server-generated filename. Artifact discard removes only the authenticated server-cache entry. Successful mutating apply and discard delete the artifact; dry-run and failed apply leave it available until expiration.
- UI sessions are host administrators. Remote clients require persisted `workspace.read` for workspace inspection, `workspace.use` for immutable session context handoff, `workspace.admin` for provider configuration/validation and workspace create/cleanup/export, and `host.apply` for dry-run or mutating host apply.
- Configure, validate, create, cleanup, export, and apply additionally require a short-lived one-time reauthentication proof bound to the authenticated principal, operation, project, canonical body hash, nonce, and expiry. The official OpenCode workspace create/remove paths are intercepted by this boundary so the generic proxy cannot bypass authorization.
- Secure Workspace settings are mutated only through `POST /api/workspaces/settings`. The route validates and canonicalizes the complete settings change before proof consumption, while the proof remains bound to the exact submitted body plus activation flag; persistence and plugin configuration use only the canonicalized copy. It then serializes settings persistence, plugin configuration, and optional OpenCode refresh. A private atomic prepared journal restores the exact prior settings and plugin entries after either a caught failure or an interrupted process; startup awaits this recovery before OpenCode can launch. Settings and OpenCode config writes publish through fsynced same-directory temporary files and atomic replacement, failing without a direct-copy fallback if replacement is unavailable. Generic settings mutation rejects every `secureWorkspaces*` key, and generic plugin-entry CRUD rejects the reserved Secure Workspace package and the actual resolved plugin filesystem identity, including explicit, normalized, and symlink-equivalent `OPENCHAMBER_WORKSPACE_PLUGIN_PATH` values.
- The generic OpenCode proxy enforces `workspace.use` for client principals whenever a workspace is explicit in query, header, or body. Session-routed calls resolve `Session.workspaceID` directly from authoritative upstream state before forwarding. Local host UI sessions remain allowed; tunnel/public UI sessions do not gain host-local authority. Direct official workspace create/remove cannot fall through to the proxy.
- OpenCode `sync-list` remains a `workspace.use` operation because it synchronizes control-plane records only; provider-resource reconciliation is the separate `workspace.admin` plus proof operation.
- Secure Workspace continuation never calls upstream warp. `POST /api/workspaces/handoffs/draft`, `POST /api/workspaces/handoffs/:operationID/commit`, `GET /api/workspaces/handoffs/:operationID`, and `DELETE /api/workspaces/handoffs/:operationID/target` are explicit OpenChamber routes registered before the proxy. They require an authenticated principal and `workspace.use`, but no admin/apply grant or password reauthentication.
- Draft extraction reads every authoritative source-message page through the official SDK and deterministically includes only visible user/assistant text and a safe inline session goal. It is bounded to 64,000 characters, rejects sources above 10,000 messages, excludes credential-like text and all non-text/hidden parts, and reports stable omission and warning codes. Localized warning prose belongs to the UI. The source boundary hashes message/part identity and text without writing source text to logs.
- Handoff operations are principal- and source/target/project-bound journals under `workspace-handoffs/`. Draft text is returned only in the immediate draft response and is never written to the journal; inspect and restart recovery expose only draft ID/revision/hash, boundary, omission codes, binding, and operation state. Format-version 1 text-bearing journals are removed during startup pruning. The directory is `0700`; files are `0600`; reads reject symlinks and non-regular files and use `O_NOFOLLOW` where supported; same-directory temporary writes, file fsync, atomic rename, and directory fsync protect each transition. Operations expire after 24 hours and are capped at 100 entries.
- Commit refetches the complete source through OpenCode's opaque `X-Next-Cursor`, deterministically re-extracts the original draft hash, and rejects a stale review if either the source boundary or extraction differs. It requires authoritative source idle state (OpenCode omits idle sessions from the status map), a canonical persisted project, authoritative source/target workspace identity, same-project routing, and connected target status. It creates a new target session with operation lineage metadata and inserts exactly one deterministic-ID text part with `noReply: true`. Create/insert timeouts inspect complete cursor-paginated target messages before retry, including when later messages have pushed the context beyond the first page; mismatched same-ID bodies fail. Exact text hash verification precedes completion and UI navigation.
- The source session is never mutated, archived, deleted, aborted, or reassigned. Exact history transfer is intentionally unsupported. Failure deletes and confirms only the newly created target session; an unconfirmed deletion persists `cleanup-required` for explicit recovery. Workspace file changes remain exclusively in export/review/apply.
- Runtime lifecycle authority rejects Secure Workspace management and handoff for remote external OpenCode hosts and reports compatibility diagnostics; loopback same-host external OpenCode remains supported. VS Code implements stable unsupported Runtime API methods and hides the workspace surface. Web, Electron, hosted mobile, and Capacitor share the server contract.

## Host Apply

- Apply accepts only a live server-side artifact and selected server-issued file/hunk IDs. Expiry is checked again after acquiring the project lock. Text operations support hunk selection; binary, rename, symlink, mode, and non-text operations are indivisible. Full text selection must equal the declared result blob, while partial selection is reconstructed from the exact baseline bytes and its staged bytes are verified against that deterministic result.
- Logical source and destination paths must be pairwise distinct and non-overlapping across the artifact; a rename cannot target its own ancestor or descendant. Directory operations participate in this path graph explicitly, and an existing directory baseline must be empty because artifact directory entries do not carry a recursive content hash.
- A per-project cross-process lock serializes mutation. Stale locks are reclaimed only when their owning process is no longer alive.
- Conflict checks compare current host entry type, mode, content hash, size, and symlink target with the artifact baseline. This is independent of Git and preserves unrelated dirty Git state.
- Selected text results and whole logical operations are materialized in a private staging tree and fingerprinted against their deterministic expected bytes, type, mode, and target. Relative paths, parent symlinks, and symlink targets are checked against root escape.
- Before mutation, every affected entry is fingerprinted, copied to backup, fsynced where Node and the host filesystem support it, and fingerprinted again. A durable journal and its directory are fsynced. Immediately before each removal/replacement, the baseline fingerprint and parent chain are revalidated; each resulting path is verified after replacement and all selected final fingerprints are verified before the committed state is written.
- Journals are versioned and scoped below a hash of the canonical project root, so an unrelated project's corrupt journal does not block this project. Recovery strictly validates the journal root, backup location, paths, and version before recursive changes. It verifies a committed final state before cleanup; an incomplete or mismatched transaction is rolled back and every restoration is fingerprint-verified. An incomplete rollback remains journaled and blocks later mutation because recovery is attempted before each apply.
- Node does not expose a portable fd-relative `openat`/`renameat` no-follow transaction API. Parent and entry checks are therefore repeated at the narrowest practical points, but a hostile local process with write access can still race in the interval between the final check and a path-based filesystem call; filesystem permissions remain the boundary for that residual TOCTOU risk.
- Successful apply consumes the artifact. Dry-run and failed apply leave it available until expiration.
