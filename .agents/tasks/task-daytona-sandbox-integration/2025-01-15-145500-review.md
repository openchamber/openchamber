# Daytona sandbox orchestration for session-isolated OpenCode containers (v2)

This change adds a Daytona integration that provisions an isolated container per chat session, running OpenCode inside each sandbox. The backend Express server orchestrates the full lifecycle — creation via the Daytona SDK, inactivity-based auto-destruction (10 minutes), SSE/HTTP proxying, and WebSocket relay to OpenCode in each sandbox. The frontend drives sandbox mode toggling, tracks per-session status, and provisions a sandbox on session creation. All seven issues from v1 have been addressed: status enum fixed to `'running'`, response fields added, auth middleware applied, dual provisioning eliminated, ws-bridge removed, openCodePort made configurable, and the WebSocket error handler added early.

**Watch for:** The `DOCUMENTATION.md` still references the removed `ws-bridge.js` module and a `bridge` property on the service object that no longer exists (confirmed). The `GET /sandbox/:sessionId/status` route returns the registry's internal `'active'` value rather than mapping it to `'running'`, so the frontend status indicator still falls through to the error state on that endpoint (confirmed). The per-request `createProxyMiddleware` instantiation in the generic proxy route allocates a new middleware on every request (confirmed) — worth evaluating whether this creates measurable overhead under load.

## High-level view

The backend composes config, registry, lifecycle, and inactivity monitor into a `createDaytonaService()` factory, conditionally enabled when `DAYTONA_API_KEY` is present. Authentication is applied to both management and proxy routes via the `uiAuthController` when enabled.

The v1 status enum fix was applied to the `POST /api/daytona/sandbox` response but not to the `GET /sandbox/:sessionId/status` route, which still returns the raw registry value `'active'`. The frontend `SandboxStatusIndicator` has no mapping for that value and falls through to the error display.

`DOCUMENTATION.md` was not updated after removing `ws-bridge.js` — it documents APIs and files that no longer exist, creating confusion for anyone onboarding to this module.

## Internal registry status vs. API response status

The sandbox registry sets `status: 'active'` on creation, while the `POST /api/daytona/sandbox` route explicitly returns `status: 'running'` in the response JSON. The `GET /sandbox/:sessionId/status` route returns `entry.status` directly from the registry — which will be `'active'`, not `'running'`:

```js
// routes.js GET /sandbox/:sessionId/status
return res.json({
  // ...
  status: entry.status, // 'active' from registry
});
```

The `SandboxStatusIndicator` falls through to the error config for unknown status strings. Either map `entry.status` to the frontend enum in all routes, or change the registry to store `'running'` from the start.

## Per-request proxy middleware instantiation

In `proxy.js`, the generic OpenCode API proxy route creates a new `createProxyMiddleware` on every request because each sandbox has a different target URL:

```js
app.use('/api/daytona/sandbox/:sessionId/opencode', authMiddleware, resolveSandbox, (req, res, next) => {
  const proxy = createProxyMiddleware({
    target: daytonaSandbox.openCodeUrl,
    // ...
  });
  proxy(req, res, next);
});
```

`http-proxy-middleware` internally creates an `http-proxy` instance with event listeners, a router, and configuration parsing. The SSE and message routes handle the hot paths directly, so this only matters if the generic proxy sees sustained traffic (filesystem operations, session list polling). A `Map<string, middleware>` cache keyed by `openCodeUrl` with eviction on sandbox destruction would eliminate repeated allocation.
