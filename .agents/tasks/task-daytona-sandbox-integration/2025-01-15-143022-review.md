# Daytona sandbox orchestration for session-isolated OpenCode containers

This change adds a Daytona integration that spawns an isolated container per chat session, running OpenCode inside each sandbox. The backend Express server orchestrates the full lifecycle â€” creation via the Daytona SDK, inactivity-based auto-destruction (10 minutes), WebSocket proxying, and SSE/HTTP forwarding to OpenCode in each sandbox. The frontend (React + Zustand) drives sandbox mode toggling, tracks per-session status, and auto-provisions a sandbox on session creation. A separate WebSocket bridge module provides a reconnecting relay, and http-proxy-middleware handles generic API pass-through.

**Watch for:** Backend returns status `'active'` but the frontend TypeScript union expects `'running'` â€” the status indicator will never show the green "Sandbox Active" dot (confirmed). The create-sandbox response omits `sessionId` and `createdAt` fields the frontend deserializer reads, producing `undefined` values in the Zustand store (confirmed). Daytona management routes have no authentication middleware, so any network-reachable caller can create/destroy sandboxes (confirmed). The WsBridge module in `ws-bridge.js` is wired into the service but never actually used by any route or proxy â€” dead code in this PR (confirmed).

## High-level view

The backend service composes five modules â€” config, registry, lifecycle, inactivity monitor, and WS bridge â€” into a single `createDaytonaService()` factory. The factory is conditionally enabled at server startup when `DAYTONA_API_KEY` is present. The inactivity monitor sweeps every 30 seconds and destroys sandboxes that exceed the configured timeout. The proxy layer (SSE, POST, generic http-proxy-middleware, and WebSocket upgrade) routes frontend traffic to the correct sandbox by sessionId path parameter, resetting the activity timer on every proxied request. Graceful shutdown destroys all active sandboxes on server stop.

The frontend integration lives in a Zustand store (`useDaytonaSandboxStore`) that maps session IDs to sandbox metadata, a `useSandboxSession` React hook that provisions the sandbox and runs a 2-minute heartbeat, and a settings panel with a "Destroy All" button. The `session-actions.ts` module also triggers sandbox provisioning on `createSession`, creating a dual-path problem with the hook â€” both attempt to provision for the same session.

There is a contract mismatch between the backend JSON response and the frontend TypeScript interface: the response body from `POST /api/daytona/sandbox` is missing fields (`sessionId`, `createdAt`) and sends a status value (`'active'`) outside the frontend's expected union type. This means the status indicator component will never render the "running" state from a fresh create.

The Daytona routes are registered on the Express app without inheriting or applying any of the existing UI-auth or tunnel-auth middleware, making the sandbox management endpoints open to anyone who can reach the server's HTTP port. When the server is bound to a network interface (LAN or tunnel mode), this is a direct escalation path: an attacker can provision and destroy sandboxes at will, consuming Daytona API quota and potentially DoS-ing sessions.

## Status and response contract mismatch

The `POST /api/daytona/sandbox` route in `routes.js` returns:

```js
return res.status(201).json({
  sandboxId: result.sandboxId,
  status: 'active',
  openCodeUrl: result.openCodeUrl,
});
```

The frontend `api.ts` declares `CreateSandboxResponse` with fields `sessionId`, `status`, `openCodeUrl`, and `createdAt`. Two problems: (1) `sessionId` and `createdAt` are absent from the response, so `data.sessionId` and `data.createdAt` are `undefined`; (2) the status value `'active'` is not in the `SandboxInfo['status']` union (`'creating' | 'running' | 'stopping' | 'destroyed' | 'error' | 'timed-out'`). The `as SandboxInfo['status']` cast hides this at compile time, but the `SandboxStatusIndicator` component matches on `statusConfig['active']` â€” which is `undefined` â€” so it falls through to the `?? statusConfig.error` fallback. Every successfully-created sandbox displays as "Error" rather than "Sandbox Active".

## Missing authentication on sandbox management routes

The existing server applies UI password auth through `uiAuthController.requireAuth` on routes registered via `bootstrapRuntime.setupBaseRoutes`. The Daytona routes are registered directly on `app` via `app.use('/api/daytona', router)` without any auth middleware. This means:

- `POST /api/daytona/sandbox` â€” create arbitrary sandboxes (costs Daytona quota)
- `DELETE /api/daytona/sandbox/:sessionId` â€” destroy any active sandbox
- `GET /api/daytona/sandboxes` â€” enumerate all active sessions
- `POST /api/daytona/sandbox/:sessionId/activity` â€” keep a sandbox alive indefinitely

When the server is exposed (LAN bind, tunnel, relay), these endpoints are reachable without credentials. The proxy routes (`/api/daytona/sandbox/:sessionId/event`, `/message`, `/opencode/*`) also lack auth.

## Dual provisioning race

`session-actions.ts` calls `provisionSandboxForSession(session.id)` immediately after `createSession` returns. Separately, the `useSandboxSession` hook fires its `useEffect` when `sessionId` changes, which also calls `createDaytonaSandbox`. The hook has a `creatingRef` guard, but it only prevents re-entry within the hook's own closure â€” it does not coordinate with the eager call in `session-actions.ts`. The result: two concurrent `POST /api/daytona/sandbox` requests for the same session. The backend's `lifecycle.createSandbox` handles this gracefully (returns existing if already registered), but there is a window where both requests hit `client.create()` in parallel before either registers, potentially creating two Daytona containers for one session. Only the second one gets registered; the first leaks.

The simplest fix is to remove the provisioning from one call site. Since `session-actions.ts` fires synchronously on session creation (before the hook mounts), keeping it there and removing the hook's auto-create logic avoids the race entirely.

## Unused WsBridge module

`ws-bridge.js` exports `connect`, `disconnect`, `relay`, and `isConnected`. The `createDaytonaService` factory instantiates it and exposes it as `bridge`. However, no route or proxy calls `bridge.connect()` or `bridge.relay()`. The only usages of `bridge` are:
- `routes.js` line 63: `daytonaService.bridge.disconnect(sessionId)` on DELETE
- `routes.js` line 82: `daytonaService.bridge.isConnected(sessionId)` in status response

Neither of these ever establishes a connection, so `isConnected` always returns `false` and `disconnect` is a no-op. The WebSocket proxying is handled entirely by `attachDaytonaWsProxy` in `proxy.js`, which creates its own upstream connections per-request. The WsBridge module appears to be an unused abstraction â€” either dead code from an earlier design, or an incomplete future feature.

## Hardcoded OpenCode URL construction

`lifecycle.js` constructs the sandbox OpenCode URL:

```js
const openCodeUrl = `https://${sandboxId}-4096.${new URL(config.apiUrl).hostname}`;
```

This assumes Daytona's proxy convention and that OpenCode listens on port 4096 in the sandbox image. There is no health check or readiness probe after `client.create()` returns. If the sandbox is still booting (container pull, OpenCode startup), the first proxy request will fail with a 503 from the upstream. The SSE proxy handles this with a 503 response, but the frontend has no retry-on-startup logic â€” it just shows "Error". A readiness poll between `create` and registration (or a retry loop in the proxy) would make startup more robust.

## WebSocket proxy edge case on upstream connect failure

In `proxy.js` `attachDaytonaWsProxy`, the upgrade handler calls `wss.handleUpgrade` and then immediately creates an upstream WebSocket:

```js
wss.handleUpgrade(req, socket, head, (clientWs) => {
  try {
    upstream = new WebSocket(upstreamWsUrl);
  } catch (error) {
    clientWs.close(1011, 'Upstream connection failed');
    return;
  }
  // ...
```

If `new WebSocket()` throws (invalid URL, constructor error), `clientWs.close()` is called. But there is also no `'error'` handler on the clientWs at this point â€” if the close frame cannot be sent (socket already half-closed during the upgrade), the error is unhandled. This is a minor edge case but can produce unhandled-exception crashes in production.

## Test coverage gaps

The 38 tests cover config resolution, sandbox registry CRUD, inactivity monitor sweep logic, and HTTP route request/response shapes. What is not tested:

- `lifecycle.js` â€” no tests for the Daytona SDK integration (create, delete, destroyAll). The SDK import makes this harder to unit-test without mocking the module.
- `proxy.js` â€” no tests for SSE forwarding, message proxying, generic API proxy, or the WebSocket upgrade handler. These are integration-heavy but the SSE backpressure and abort handling have subtle edge cases.
- `ws-bridge.js` â€” no tests (and as noted, also unused).
- Frontend: no tests for `useSandboxSession` hook, `session-actions.ts` provisioning, the store, or the settings component.
- The dual-provisioning race between `session-actions.ts` and `useSandboxSession` has no integration coverage.
